# 009-portfolio-risk-hedge-decision-layer.md

---

## 1. Milestone

Transitioned from portfolio exposure analysis to designing a **decision-layer system** that maps risk exposures to **user-selectable hedge strategies**, moving beyond static recommendations into interactive portfolio construction logic.

---

## 2. Key Requirements Introduced

- Portfolio analysis should not output generic “add X sector” suggestions.
- Hedge recommendations must be **choice-driven**, not prescriptive.
- System should:
  - Identify **macro risk exposures** (e.g., inflation, rates, dollar).
  - Present **multiple hedge strategies per risk**.
  - Allow the user to **select a preferred hedge approach**.
- Portfolio report should:
  - Explain current structure (factor + macro alignment).
  - Highlight **specific vulnerabilities**.
  - Frame recommendations as **tradeoffs**, not solutions.
- Sector-level interpretation must consider **functional roles** (e.g., healthcare ≠ always defensive).
- Strategic lens (e.g., **state capitalism / energy importance**) should influence recommendations.

---

## 3. Design Decisions

### 3.1 Shift from Allocation-Based to Risk-Based Model

- Move from:
  - “Add missing sectors”
- To:
  - “Identify risk → map to hedge options”

```ts
type RiskFactor =
  | "inflation"
  | "interest_rates"
  | "economic_slowdown"
  | "dollar_weakness";
```

---

### 3.2 Hedge Strategy Abstraction

- Introduce **many-to-one mapping**:
  - One risk → multiple hedge strategies

```ts
type HedgeStrategy = {
  id: string;
  targetRisk: RiskFactor;
  label: string;
  description: string;
  tradeoffs: string[];
  factorImpacts: Record<string, number>; // tag-level adjustments
};
```

---

### 3.3 Decision Layer (Core Engine)

- New system boundary:
  - Between **analysis layer** and **portfolio construction**

```ts
type Recommendation = {
  risk: RiskFactor;
  exposureLevel: number;
  message: string;
  hedgeOptions: HedgeStrategy[];
};
```

---

### 3.4 Functional Sector Decomposition

- Reject naive sector assumptions:
  - Healthcare split into:
    - Defensive (pharma)
    - Growth (healthcare SaaS)

```ts
type SectorRole =
  | "defensive"
  | "growth"
  | "inflation_hedge"
  | "rate_sensitive";
```

---

### 3.5 Portfolio Interpretation Layer

- Structured output sequence:

1. Portfolio Structure Summary  
2. Risk Exposure Identification  
3. Hedge Decision Prompt  
4. Strategy Options (tradeoff-based)

---

### 3.6 Non-S&P Design Philosophy

- Explicitly avoid convergence to S&P500:
  - Maintain **intentional overweight (AI)**
  - Add **minimal hedges for survivability**

---

## 4. Iterations / Corrections

- Issue: Initial recommendation logic suggested “add all missing sectors”.
- Fix: Reframed as **risk → hedge selection model**.
- Result: Avoids S&P-like convergence; enables intentional portfolio shaping.

---

- Issue: Healthcare treated as uniformly defensive.
- Fix: Split into **pharma (defensive)** vs **healthcare growth (rate-sensitive)**.
- Result: More accurate risk mapping; prevents false hedging assumptions.

---

- Issue: Recommendations were prescriptive (“add energy”).
- Fix: Converted to **user choice framework** (energy vs financials vs non-US).
- Result: Introduced decision-layer UX with explicit tradeoffs.

---

- Issue: Portfolio appeared diversified but had hidden concentration (AI + US + megacap).
- Fix: Introduced **factor-level risk interpretation**.
- Result: Surfaced implicit macro bet (low rates + US dominance).
