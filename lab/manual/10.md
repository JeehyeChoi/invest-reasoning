## 1. Milestone

Transition from portfolio-focused logic to a structured **SEC-based data ingestion + factor computation architecture**, including database schema design, workflow restructuring, and separation of concerns between data fetching, storage, and computation.

---

## 2. Key Requirements Introduced

- SEC `companyfacts` should be the **primary raw data source** (not FMP).
- Raw data must be stored in a **source-aligned schema** (not pre-processed).
- Introduce **factor-based system** with:
  - 20 factors
  - 3 axes (fundamentals, etf, narrative)
- Separate:
  - **fetch (client)** vs **get (service)** vs **compute (workflow)**
- Support **bulk ingestion (SEC nightly)** + **per-ticker fallback**
- Workflow must be refactored into **clear step-based structure**
- Naming consistency across:
  - schemas
  - services
  - workflows
- Ensure **reusability of ticker metadata (CIK)**

---

## 3. Design Decisions

### Data Model

- Introduced table:

```sql
sec_company_fact_points
```

- Stores:
  - taxonomy
  - concept
  - unit
  - fact point fields (value, period, form, etc.)
  - raw JSON (`raw_point_json`, `raw_concept_json`)

- Design principle:
  - **source-aligned (SEC companyfacts)**
  - not pre-aggregated (no revenue-only table)

---

### Abstractions

- **Client layer**
  - `fetchFromSec`
  - `fetchSecCompanyFacts`
  - `fetchSecDocument`

- **Service layer**
  - `getRevenueFacts` (fetch orchestration only)
  - no computation

- **Workflow layer**
  - `computeRevenueStep`
  - `computeFactorScoresStep`
  - `buildSnapshotStep`
  - `persistSnapshotStep`

---

### System Boundaries

- Clear separation:

```text
client → fetch (external API)
service → get (data retrieval / normalization)
workflow → compute (business logic)
db → storage (raw + computed)
```

- Portfolio analysis workflow separated from:
  - factor computation
  - LLM request logic

---

### Workflow Structure (Refactored)

```text
steps/
  growth/
    fundamentalsBased/
      computeRevenueStep.ts

  computeFactorScoresStep.ts
  buildSnapshotStep.ts
  persistSnapshotStep.ts
```

- Axis-first computation logic:
  - axis → factor → metric

---

### Data Ingestion Strategy

- Hybrid ingestion model:

```text
bulk (daily SEC ZIP) + per-ticker fetch fallback
```

- Bulk:
  - nightly snapshot (~3 AM ET)
- API:
  - near real-time

---

### Naming Conventions

- `fetch` → external call (client)
- `get` → service-level retrieval
- `compute` → calculation logic
- `load` → DB read

## 4. Iterations / Corrections

### • Issue:
Inconsistent use of `fetch` vs `get` across services

### • Fix:
Defined strict rule:
- client = fetch
- service = get

### • Result:
Clear separation of responsibilities

---

### • Issue:
`fetchFilingDocument` implemented in services instead of client

### • Fix:
Moved to `clients/secDocument.ts`

### • Result:
Consistent API boundary

---

### • Issue:
Ticker metadata function too narrow (`getTickerCiksByTickers`)

### • Fix:
Replaced with:

```ts
getTickerProfilesByTickers
```

### • Result:
Reusable abstraction for all workflows

---

### • Issue:
Factor workflow naming too vague (`computeTickerFactorInputStep`)

### • Fix:
Refactored into explicit step hierarchy

### • Result:
Readable and scalable workflow structure

---

### • Issue:
Confusion between portfolio-analysis and factor computation

### • Fix:
Separated workflows:
- portfolio-analysis
- ticker-factor-snapshot

### • Result:
Clear system boundaries

---

### • Issue:
Uncertainty about SEC bulk vs API usage

### • Fix:
Adopted hybrid ingestion strategy

### • Result:
Production-aligned architecture
