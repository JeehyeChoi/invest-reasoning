# 1. Milestone

Set up GitHub repository integration, establish a repeatable Git workflow, and resolve commit author identity issues while introducing proper `.gitignore` handling.

---

# 2. Key Requirements Introduced

- Initialize and connect a local project to a GitHub repository.
- Enable continuous updates (push workflow).
- Choose a concise, meaningful repository name (`invest-reasoning`).
- Ensure secure authentication (PAT, avoid storing tokens in files).
- Fix incorrect commit author identity.
- Prevent unwanted files (e.g., `.tar.gz`, `.env`) from being tracked.

---

# 3. Design Decisions

- **Repository Naming**
  - Use a shorter, meaningful name (`invest-reasoning`) over verbose alternatives.

- **Workflow Design**
  - Standard Git workflow:
    - `add → commit → push`
    - Use `-u` for upstream tracking.

- **Authentication Strategy**
  - Use HTTPS + PAT with macOS Keychain for credential storage.
  - Avoid storing tokens in `.env` or files.

- **Identity Handling**
  - Use Git config (`user.name`, `user.email`) aligned with GitHub account.
  - Prefer resetting history (re-init) over complex rebase for early-stage projects.

- **History Management**
  - Chose full reset (`rm -rf .git`) instead of rebase for simplicity and reliability.

- **File Tracking Strategy**
  - Use `.gitignore` for future exclusions.
  - Use `git rm --cached` to stop tracking existing files.

  - Fix: Introduced PAT-based authentication and Keychain usage
  - Result: Secure and persistent authentication setup

- Issue: Commit author incorrectly set to `Joy` / `dropwell`
  - Fix: Updated Git config and reset repository history
  - Result: Clean commit history with correct identity

- Issue: `git rebase` errors (`HEAD~3`, unstaged changes)
  - Fix: Abandoned rebase approach in favor of full repo reset
  - Result: Simplified resolution path

- Issue: Force push rejected with `--force-with-lease`
  - Fix: Used `--force` due to fresh repo initialization
  - Result: Successful overwrite of remote history

- Issue: Confusion about `.gitignore` behavior
  - Fix: Explained difference between tracked vs untracked files and use of `git rm --cached`
  - Result: Correct mental model of Git tracking system
