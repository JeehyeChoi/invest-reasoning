# 1. Milestone

Clarified how to move from GitHub–Vercel integration to an actually running deployed app, and how to handle API-key-dependent features in a production deployment context.

---

# 2. Key Requirements Introduced

- Understand how deployment is triggered after GitHub–Vercel integration.
- Ensure the app can run even when API keys are not provided.
- Avoid exposing API keys publicly (especially in GitHub).
- Maintain a deployable, demo-ready state for portfolio purposes.

---

# 3. Design Decisions

- **Deployment Model**
  - Use Vercel’s Git-based auto-deployment (push → deploy).
  - No manual “run” step; deployment is event-driven.

- **Environment Variable Strategy**
  - Separate local `.env.local` from Vercel environment variables.
  - Store all secrets (API keys) only in Vercel dashboard.
  - Support environment-specific configs (Production / Preview / Dev).

- **Runtime Behavior Abstraction**
  - Introduce conditional execution based on presence of API keys:
    - `hasOpenAIKey` style boolean gate.
  - Separate:
    - External API-dependent logic
    - Fallback/mock logic

- **System Boundary Definition**
  - External APIs (OpenAI, market data) are optional dependencies.
  - Core app (UI, routing, basic logic) must run independently.

- **Resilience / Demo Strategy**
  - Implement "demo mode" when env variables are missing.
  - Prevent runtime crashes due to missing keys.
  - Ensure recruiter-facing stability.

# 5. Artifacts Produced

- **Environment variable handling pattern**

```ts
const hasOpenAIKey = !!process.env.OPENAI_API_KEY;

export async function POST() {
  if (!hasOpenAIKey) {
    return Response.json({
      mode: "demo",
      analysis: "This is a sample portfolio analysis for demo deployment.",
    });
  }

  // actual API call
}
```

- **Deployment mental model**
  - Git push → automatic Vercel deployment

- **Fallback logic definition**
  - API present → real execution
  - API absent → mock/demo response
