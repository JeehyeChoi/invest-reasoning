# 012-sec-series-reconciliation-and-factor-extension.md

## 1. Milestone

Refactor SEC company facts series reconstruction logic (quarterly flow reconciliation) and extend the factor system from revenue to net_income while identifying architectural separation issues (series vs factor vs display).

---

## 2. Key Requirements Introduced

- Quarterly reconstruction must be **deterministic based on annual anchors** (start/end alignment).
- Support reconstruction from:
  - direct 3M
  - 6M cumulative
  - 9M cumulative
  - trailing 6M
- Must handle **partial data cases** (e.g., only Q3 exists, or only cumulative data exists).
- Do not rely on fragile positional assumptions (`sortedDirect[0]`, etc.).
- Ensure **direct 3M data is always preserved as baseline**.
- Add support for **net_income metric across full pipeline** (series → factor → display).
- Fix frontend issue where **formula + interpretation not showing for net_income**.
- Improve performance:
  - single-pass classification
  - avoid repeated date parsing
- Database uniqueness constraint must include **(start, end)**.

---

## 3. Design Decisions

### Data Model

- `sec_companyfact_series` uniqueness updated to:
```sql
UNIQUE (cik, metric_key, period_type, display_frame, start, end)
```

- `COMPANY_FACTS_SERIES_TAG_META`
  - remains the **source of truth for tag → metricKey mapping**

- Metric definitions centralized in:
  - `src/backend/schemas/sec/metrics.ts`
  - `src/backend/schemas/factor.ts`

---

### Series Reconstruction Architecture

- Core function:
```ts
buildAnnualCycleRows(...)
```

- Anchor-based inference:
  - Q1 → matches `annual.start`
  - Q4 → matches `annual.end`
  - Q2/Q3 → inferred via:
    - cumulative6m
    - cumulative9m
    - trailing6m
    - relative position

- Reconstruction phases:
  1. Direct classification (anchor-based)
  2. Reconstruction using cumulative deltas
  3. Final fallback completion

- Critical rule:
  > direct 3M = baseline, reconstruction = augmentation (not replacement)

---

### Factor System

- Factor pipeline expanded:
  - revenue → net_income

- Shared logic extracted:
  - `normalizeGrowthMetrics`
  - `interpretGrowthMetrics`

- Factor identity structure:
```
factor + axis + metricKey
```

- Blueprint:
```ts
FACTOR_BLUEPRINTS
```

---

### System Boundary Separation

Explicit separation established:

| Layer    | Responsibility |
|----------|--------------|
| series   | raw → structured financial time series |
| factor   | compute metrics & scores |
| display  | UI representation (formula, interpretation) |

## 4. Iterations / Corrections

### 1. Quarter classification logic incorrect

- Issue:
  - Used positional logic (`find`, `sortedDirect[0]`)
- Fix:
  - Switched to **annual anchor-based classification**
- Result:
  - Deterministic quarter identification

---

### 2. Reconstruction failed when partial data present

- Issue:
  - Missing quarters caused entire year to drop
- Fix:
  - Introduced fallback reconstruction from cumulative data
- Result:
  - Partial datasets now usable

---

### 3. Direct 3M overwritten by reconstruction

- Issue:
  - Reconstruction logic replaced valid data
- Fix:
  - Treat direct data as baseline, reconstruction only fills gaps
- Result:
  - Data integrity preserved

---

### 4. Sorting crash (`localeCompare` error)

- Issue:
```ts
(a.start ?? "").localeCompare is not a function
```
- Fix:
  - Normalize to string before compare
- Result:
  - Stable sorting

---

### 5. Over-complex conditional logic

- Issue:
  - Deep `if / else if` chains with unclear priority
- Fix:
  - Reorganized into:
    - classification phase
    - reconstruction phase
- Result:
  - More predictable execution order

---

### 6. Missing interpretation/display for net_income

- Issue:
  - `getTickerOverview` only handled revenue
- Fix:
```ts
metricKey === "revenue" || metricKey === "net_income"
```
- Result:
  - net_income now shows formula + interpretation

---

### 7. Performance inefficiencies

- Issue:
  - multiple array scans
  - repeated `new Date()`
- Fix:
  - single-pass classification
  - lexicographic date comparison
- Result:
  - reduced computational overhead
