# 012-factor-modeling-and-robustness-roadmap.md

---

## 1. Milestone

Defined the transition point from factor analysis to modeling, establishing a roadmap for introducing scenario-based perturbation, robustness evaluation, and eventual portfolio rebalancing logic.

---

## 2. Key Requirements Introduced

- Modeling will be introduced **after factor analysis is complete**, not before.
- System should support **weight perturbation (+/- adjustments)** for:
  - factor loadings
  - scenario shocks
- Goal is not return maximization but:
  - **scenario-robust portfolio**
  - **regime-robust allocation**
- Need to identify **stable points** where:
  - small weight changes do not alter conclusions significantly
- Introduce **robustness scoring** before any optimization or rebalancing logic
- Avoid early optimization complexity; follow staged rollout

---

## 3. Design Decisions

### 3.1 Separation of Model Layers

Two independent weight systems must be modeled:

```
tag → factor (loading model)
factor → scenario (shock model)
```

- These must remain decoupled to allow independent perturbation and calibration.

---

### 3.2 Workflow Expansion Strategy

Current pipeline:

```
byTag → factorExposure → scenarioImpact
```

Planned extended pipeline:

```
byTag
→ factorExposure (loadings)
→ scenarioImpact (shocks)
→ robustnessScore
→ suggestedRebalance
```

---

### 3.3 Stability as Core Objective

Three distinct stability concepts defined:

- **Model Stability**
  - Sensitivity of conclusions to loading changes

- **Portfolio Robustness**
  - Stability of portfolio under multiple scenario shocks

- **Optimization Stability**
  - Avoiding highly sensitive or unstable allocation outputs

---

### 3.4 Phased Implementation Strategy

Strict sequencing enforced:

1. Fixed loadings + fixed scenario shocks
2. Sensitivity analysis (perturbation of weights)
3. Robustness scoring
4. Rebalancing suggestions (only after above are stable)

---

### 3.5 System Role Reframing

System is explicitly **not a prediction engine**, but:

- Assumption sensitivity analysis engine
- Scenario-based portfolio design engine
- Worldview-consistent allocation framework

---

### 3.6 Traceability Requirement

Outputs must support **contribution-level explainability**:

- ticker → factor contribution
- tag → factor influence
- factor → scenario impact

This enables robustness validation and debugging.

---

### 3.7 Configurability Requirement

Even if weights are initially hardcoded, system must allow future replacement via:

- config files
- database tables
- scenario presets

---

## 4. Iterations / Corrections

- Issue:
  Early tendency to move toward optimization prematurely

- Fix:
  Enforced staged approach: analysis → sensitivity → robustness → optimization

- Result:
  Prevented over-complexity and ensured foundational correctness first

---

- Issue:
  Treating weight as a single dimension

- Fix:
  Explicit separation into:
  - loading weights
  - scenario shock weights

- Result:
  Enabled independent calibration and proper modeling flexibility

---

- Issue:
  Ambiguity in "stable point" definition

- Fix:
  Formalized into three categories:
  - model stability
  - portfolio robustness
  - optimization stability

- Result:
  Clear evaluation criteria for future modeling steps

## 6. Open Questions / Next Steps

- Design of `robustnessScore`:
  - average scenario impact?
  - worst-case loss (min-max)?
  - variance across scenarios?
- Formal definition of perturbation ranges:
  - how much to vary loadings and shocks?
- Data structure for storing scenarios and weights:
  - DB schema vs config-based?
- Design of `runSensitivityStep` and `computeRobustnessStep`
- Criteria for triggering rebalancing suggestions
- Whether to incorporate formal optimization (e.g., min-max) or heuristic rules

- Introduced distinction between:

  - scenario shock weights
- Guided system direction toward:
  - stability-focused architecture
  - sensitivity-aware modeling
