# 007_postgresql-metadata-layer.md

---

## 1. Milestone

Transition from in-memory metadata handling to a persistent PostgreSQL-backed metadata and tagging system, establishing a stable data layer for portfolio analysis.

---

## 2. Key Requirements Introduced

- Replace in-memory/profileStore-based metadata with a database-backed system.
- Persist ticker profile data (sector, industry, etc.) instead of refetching repeatedly.
- Persist inferred tags for each ticker.
- Ensure metadata is resolved dynamically at analysis time (not precomputed globally).
- Prioritize metadata/tag infrastructure before portfolio persistence and auth.
- Avoid long-lived caching due to dynamic strategy/tag logic.
- Maintain separation between raw provider data and inferred tags.

---

## 3. Design Decisions

### Data Model Separation
- **TickerProfile (raw data)**:
  - Source: external API (FMP)
  - Stable, infrequently changing
- **TickerTags (derived data)**:
  - Source: inference rules
  - Strategy-dependent and evolving

---

### Database as Source of Truth
- Eliminated in-memory cache (`profileStore`)
- PostgreSQL becomes authoritative storage for:
  - ticker profiles
  - inferred tags

---

### Table Design

**ticker_profiles**
- Stores raw metadata from provider
- Includes:
  - ticker (PK)
  - sector, industry, region
  - market data fields
  - source + timestamps

**ticker_tags**
- Stores inferred tags per ticker
- Includes:
  - ticker (FK)
  - tag
  - source_rule
  - inferred_at
- Unique constraint on (ticker, tag)

---

### Workflow Integration

Refined metadata resolution flow:

1. Query DB for ticker profile
2. If missing → fetch from API
3. Save profile to DB
4. Infer tags from metadata
5. Upsert tags into DB
6. Return combined metadata for analysis

---

### Step Responsibility Separation

- `resolveMetadataStep`
  → data fetching + tag inference + DB persistence

- `computeExposureStep`
  → consumes metadata only (no fetching/inference)

---

### Helper Function Boundary Fix

- Removed direct `context` access inside helper functions
- Passed `metadataByTicker` explicitly as argument

---

### Tag System Architecture

- `AnalysisTag` defined as internal semantic layer (ontology)
- External sector/industry retained as raw data
- Tag inference rules remain code-based (not DB yet)

---

### Sequencing Strategy

1. PostgreSQL integration
2. Metadata + tag persistence
3. Replace in-memory store
4. Portfolio storage (later)
5. Auth (last)

---

## 4. Iterations / Corrections

- Issue: `context is not defined` inside helper function  
  Fix: Passed metadata as function argument instead of accessing context directly  
  Result: Proper separation of concerns and no runtime error

---

- Issue: Confusion about `'Consumer Defensive'` appearing in output  
  Fix: Identified as raw provider (FMP) sector value  
  Result: Clarified distinction between raw metadata vs inferred tags

---

- Issue: Uncertainty whether `AnalysisTag` system is necessary  
  Fix: Defined tags as semantic analysis layer (not raw data)  
  Result: Confirmed tags are essential for portfolio reasoning

---

- Issue: Overuse of in-memory caching (`profileStore`)  
  Fix: Decided to move persistence to PostgreSQL  
  Result: Clear data boundary and scalable architecture
