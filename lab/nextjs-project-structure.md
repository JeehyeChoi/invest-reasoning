범용 버전 폴더 설명


src/
├─ app/                    ← 프레임워크 진입점
│                          ← Next.js가 인식하는 페이지, 레이아웃, 라우트 핸들러를 둠
│                          ← “서비스가 어디서 시작되는가”를 정의하는 레이어
│                          ← 비즈니스 로직 자체보다는 연결과 진입을 담당
│
│  ├─ page.tsx             ← 기본 진입 페이지
│  │                        ← 첫 화면 또는 상위 진입 화면
│  │                        ← 여러 기능(feature)을 조합해 보여주는 역할
│  │
│  ├─ layout.tsx           ← 공통 레이아웃
│  │                        ← 여러 페이지에 공통으로 적용되는 구조
│  │                        ← 전역 스타일, metadata, navigation, provider 등을 배치
│  │
│  └─ api/                 ← 서버 API 엔트리
│                           ← HTTP 요청을 받아 내부 로직으로 전달하는 얇은 레이어
│                           ← route handler는 입출력 처리에 집중하고,
│                           ← 실제 로직은 backend로 넘기는 것이 핵심
│
├─ features/               ← 기능/도메인 단위 구성
│                          ← “기술 종류”보다 “무슨 기능인가”를 기준으로 묶는 영역
│                          ← 서비스가 커질수록 가장 중요한 구조적 중심축이 됨
│                          ← 예: auth, dashboard, search, billing, analysis
│
│  ├─ <feature-name>/      ← 특정 기능 하나를 대표하는 폴더
│  │                        ← 해당 기능의 UI, 입력 구조, 화면 가공 로직을 함께 관리
│  │
│  │  ├─ components/       ← 해당 기능 전용 UI 컴포넌트
│  │  │                     ← 그 기능 안에서만 의미 있는 화면 조각
│  │  │                     ← 범용 공용 컴포넌트와 구분됨
│  │  │
│  │  ├─ schemas/          ← 데이터 구조와 검증 규칙
│  │  │                     ← 폼 입력, 요청 바디, 응답 shape 검증 등에 사용
│  │  │                     ← “이 기능이 어떤 데이터를 다루는가”를 명확히 해줌
│  │  ├─ hooks/          ← hooks/는 React hook이 있을 때만
│  │  │                     ← services/는 UI 가공 로직이 있을 때
│  │  │
│  │  ├─ services/         ← UI 관점의 가공 로직
│  │  │                     ← 화면 표시를 위한 변환, 정리, 조립 로직
│  │  │                     ← 서버 비즈니스 로직과는 다르고,
│  │  │                     ← feature 내부에서만 의미 있는 처리에 집중
│  │  │
│  │  ├─ utils/            ← 작은 보조 함수
│  │  │                     ← 포맷팅, 단순 계산, 정렬, 문자열 처리 등
│  │  │                     ← 규모가 큰 로직을 넣는 곳은 아님
│  │  │
│  │  └─ prompts/          ← 해당 기능에 귀속된 프롬프트 자산
│  │                        ← LLM 기능이 있는 서비스에서만 필요
│  │                        ← 전역 공용이 아니라 특정 기능에 붙는 프롬프트를 둠
│
├─ backend/                ← 순수 서버 로직
│                          ← 프레임워크와 분리된 핵심 처리 로직
│                          ← app/api가 요청을 받으면 실제 처리는 여기서 수행
│                          ← 나중에 서버를 분리해도 재사용하기 쉬운 구조를 목표로 함
│
│  ├─ clients/             ← 외부 시스템 연결부
│  │                        ← DB, 외부 API, 인증 제공자, AI API, 결제 API 등
│  │                        ← 외부 의존성을 한곳에 모아 교체와 테스트를 쉽게 만듦
│  │
│  ├─ services/            ← 서버 비즈니스 로직
│  │                        ← 여러 client를 조합해 실제 기능을 수행
│  │                        ← “이 서비스가 실제로 하는 일”이 가장 많이 모이는 영역
│  │
│  └─ utils/               ← 서버 전용 보조 함수
│                           ← 에러 변환, 서버 포맷팅, 공통 헬퍼 등
│                           ← 단, 너무 많은 걸 몰아넣지 않도록 주의
│
├─ shared/                 ← 여러 레이어가 함께 쓰는 공통 자원
│                          ← 특정 feature나 특정 backend 로직에만 속하지 않는 것
│                          ← 서비스 전반의 계약과 일관성을 담당
│
│  ├─ types/               ← 공통 타입
│  │                        ← 프론트/백엔드가 함께 이해해야 하는 데이터 계약
│  │                        ← 예: User, ApiResult, Pagination, ErrorShape
│  │
│  └─ constants/           ← 공통 상수
│                           ← enum성 값, 기본 설정값, 제한값, 공통 키 등
│                           ← 하드코딩을 줄이고 의미를 분명히 함
